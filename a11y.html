Notes on Accessibility

In the UK, 1 in 5 people have a disability – this could be visual, hearing, motor or cognitive (affecting memory and thinking). But the concept of accessibility doesn’t just apply to people with disabilities – all users will have different needs at different times and in different circumstances.
For example: location (noise, slow wifi) health (tired, recovering) equipment (mobile or older browser).
Assisted digital support: helping user with low digital skills, or limited access. There are 14% in the red on the digital inclusion scale.

Making things more accessible makes things better for all users

What is Digital Accessibility?:
"Digital accessibility is the ability of a website, mobile application or electronic document to be easily navigated and understood by a wide range of users, including those users who have visual, auditory, motor or cognitive disabilities."

HTML5, CSS + Aria

HTML has historically lacked some key semantic markers, such as the ability to designate sections of the page as the header, navigation, main content and footer.
With HTML5, these designations are possible, using the new elements for it: header, nav, main, and footer.
Similar functionality is available using the ARIA (Accessible Rich Internet Application) attributes role="banner", role="navigation" and role="contentinfo"

ARIA provides a way to specify roles for documents areas and more. The available Document Landmark Roles are:

  • banner: site-orientated content, such as the name of the website, title of the page, and or logo
  • navigation: the area that contains the navigation links for the document or website
  • main: the main or central content of the doucment
  • search: this section contains the search functionality for the website
  • article: stand alone content that makes sense out of context from the rest of the document. Example might be blog posting, a comment on a blog, a forum post, etc... Specifically, a blog posting might be identified as an article and individual blog comments might also be marked up with a role of article within that blog posting
  • complementary: supporting content for the main content
  • contentinfo: informational child content, such as footnotes, copyrights, links to privacy statement, links to preferences and so on.

ARIA Live Regions - Mozilla

Screen reader support for ARIA live regions

Using the aria-role description attribute

An ARIA live region is a simple mechanism for notifying screen readers when content is updated on the page. Despite the obvious Accessible User Experience (AUX) benefits that live regions bring, screen reader support is disappointingly inconsistent.

The aria-live attribute identifies an element as a live region. It takes three possible values:

  • Off (no notification)
  • Polite (screen reader notifies user once current task is complete)
  • Assertive (screen reader interrupts current task to notify user)

Note: Assistive technologies will announce dynamic changes in the content of a live region. The live region must first be present (and usually empty), so that the browser and assistive technologies are aware of it. Any subsequent changes are then announced.

Deaf and hard of hearing people

Captioning: It’s Time for a Revolt

Sounding out the web: accessibility for deaf and hard of hearing people

What can subtitles and text do? and not do?

Subtitles or captions for hard of hearing people provide text information on the screen that is only available through sound. As well as providing a text transcription of what is being said, captions can explain which character is speaking and that, for example a telephone is ringing or a dog is barking. Text cannot however convey the tone, pitch and intonation of a voice

TV captioning

Only two lines of text are provided for TV captions and so when people are speaking very quickly the text may disappear from the screen too quickly to be able to read

TV captioners therefore often simplify the captions rather than present verbatim captions. Pre-prepared block captions can be used for recorded videos, whereas scrolling captions are used for live captioning which are harder to read

Highly trained captioners using chorded phonetic keyboards are necessary for verbatim captioning of very fast speech (e.g. 200 words per minute), whereas repeating and simplifying what is being said using speech recognition can be used for slower speech rates

Accuracy measures normally simply count the number of errors in captions without taking into account the relative importance of the words in conveying meaning. This makes it very difficult to measure the accuracy of simplified captions and compare them with verbatim captions

Web captioning

On the web it is possible to provide many more lines of text than on a TV and also to automatically identify which word is being spoken through underlining or highlighting

Foreign language subtitles

Foreign language subtitles are used to present a text translation into a target language from the language being spoken in the video. It is cheaper than providing the spoken soundtrack in the target language

Sign language

Sign language is required by Deaf viewers whose first language is sign language and who may find reading captions too difficult. Every country has its own sign language

Audio description

Audio description is used to provide a spoken description of important information that is only available visually

This information is presented when there is no other information being presented through speech or sound and so the short time available often means that only descriptions that are absolutely essential to understanding can be provided

Web systems are available that pause the video to allow extra time for the audio description

When listening to an audio with no video then all the spoken information required can be provided by a transcript. It is much faster to read a transcript than listen to spoken information presented at normal speed

Unlike captions, a transcript for a video however, cannot convey which text corresponds with which sounds and which images

Captions are also very popular with many people who have no disability, as they can help with the understanding of what is being said for example when there is a low sound level and/or a high background noise level

There can be different regulations for captions or audio descriptions for broadcasters in different countries for both TV and internet. These may vary in terms of the accuracy, number of hours, types of broadcasts and the times they are shown

Transcript

They are really good as it's easy to use of the broadband is not great (compare to a video) you can search it and it's quicker to read it than to listen to it

Channel favoured by deaf users:

  • Live text chat is often the preferred option (also for autistic users)
  • Video relay service such as Signvideo, which allows users to make and receive BSL interpreted video calls
  • Textphone
  • Email but they usually don't know how long it will take to receive a response

What’s the best dyslexia font?

Dyslexia Font and Style Guide

Dyslexia type face

Some contrast levels of print on different backgrounds, font styles and size of print, plus the overall look and feel may cause increased reading difficulties.

Some individuals find the option to change screen colours with a virtual overlay or via a different background colour may help. Both these changes can be achieved through the use of bookmarks, plugins or toolbars in browsers.

The position of the text on the screen is also important - avoiding clutter and making good use of white space to delineate particular points in the content can assist all readers. In English text that is left justified with a ragged right edge allows the reader to keep an eye on where they have reached on a long page or text, whereas justified text such as may be seen in a newspaper column, provides no such guide.

Some guides also suggest restricting the width of lines to around 60-70 characters such as the updated BDA Style Guide which is backed by research, although some people find even narrower columns can help increase reading speeds.

Individual preferences need to be considered and there are readers who find narrow fonts with very little character spacing hard to cope with although research has shown this is not so much of an issue as line spacing. However, choosing a sans serif font, such as Arial or Verdana with increased letter spacing, can increase readability generally.

10 Principles of Accessibility for Web and Mobile Design

  • blind gets 80% of accessibility issues (includes low vision and colour blindness)
  • alt text (don't need to say 'image' and only say what is relevant (also important to SEO) - keep it to 50 characters long max, if not need then do" alt="" so screen readers ignore them. Axess Lab guidance about alt text
  • hamburger menu are usually not accessible
  • content placed 'out of the way' won't be found or seen
  • accessibility testing with users can reveal huge gaps (don't just use checkers)
  • disabling zoom on mobile makes it inaccessible
  • accessibility is easily learned (to code for it) and cheap if done up front
  • accessibility doesn't mean ugly - visual bias is your worse enemy
  • check mobile accessibility separately (including responsive issues (collapse/expand can be lost, especially with magnification)
  • embrace the access attitude: people first, clear purpose with well defined goals, solid structure (standards), easy interaction, helpful wayfinding, clean presentation, plain language (create da conversation), accessible media

Accessibility impact SEO: the better the experience (time on site) and less barriers is considered good by algorithms (like google)

Trying to navigate a site after a poor night sleep, or after a beer can put you in a good place to test how cognitively impaired people might be ;)

checkers are good but do not replace testing: constrast (contrast checker tool WCAG 2), colour blind checkers (Vischeck). Sometimes, they raise issue that are not that important so you don't need to fix all.

WCAG Web Content Accessibility Guidelines

  • perceivable: information and user interface components must be presented to users in ways they can see or hear
  • operable: UI components and navigation must be designed so that users can interact with them and they can support assistive technologies such as screen readers
  • understandable: information and the operation of the UI must communicate clearly and consistently so that the content is readable
  • robust: content must be written so that it can be interpreted reliably by a wide variety of user agents, including assistives technologies

Compliance requirement A, AA, AAA

AA is the standard most used. A is low (essential for basic accessibility), AA is moderate and AAA is the highest (Gold standard).

Inclusive Design strategies and tactics

AT= Assistive Technology or Access Technology

Inclusive Design is an ideal approach for designing for universal access. To get it right, you need to incorporate real people in the design, including:

  • ensuring that everyone involved in the web projects understands the basics of how people with disabilities use the web
  • involving users with disabilities early enough and throughout the design process
  • involving users in evaluating web accessibility

Disability personas

  • what devices are they confortable with?
  • how familiar are they with the domain, AT, device?, experiences, skills with the relevant tools and frequency of use
  • do they have multiple disabilities?
  • to what degree is the disability?, nature of the limitation
  • we don't need so much info about demographichs (age, gender etc) but more about their pain points, behaviours and challenges
  • they come from field studies

User Journey

  • identify user journey (key tasks / use cases / user stories
  • apply your guidelines to them (tasks vs guidelines)
  • simulations workshop is good, but pretending to have the disability is not the same as living with it

Accessibility needs to be built into Design and Dev from the start. Bake it in up front in 3 steps:

  • build an Accessibility Program (at design phase, at dev phase and plan for post-launch support, maintenance and updating) work it into product management, UX design, code review and QA
  • build a library of Accessible components and patterns
  • build a library of accessibility Test Cases
  • make Accessibility part of your vision and your process: design, dev, testing, deployment, support, modification to system
  • develop a roadmap
  • use the IT Accessiblity Playbook - 12 rules developed byt the Access Board

Section 508 Playbook

Build the business case ROI

  • SEO benefits
  • increased Usability from accessibility
  • remove Accessibility bug fixing time
  • support for aging population 51% of US seniors are on broadband (speed of site, a bounce factor)
  • reduce legal risk
  • access to contracts (sell-ability)
  • HTML5 efficiency / responsive web design

Team roles - all hands on

  • product/program managers: integrate requests from the start, develop business case, track progress toward access goals
  • UX Designers, UR - conduct field studies + testing + A'QA'
  • developers - clean code (use ARIA), checkers, QA, ACC Code libraries
  • QA team - build from start, unit testing, test against ACC user stories

Write user stories for accessibility, for example:
As a screen reader user who is blind
I WANT headings to be identified as actual headings
SO THAT I can navigate the web page by heading and understand the page's structure.

Triangulated approach: use all the methods together: checkers, guidelines (WCAG), user access testing and retest user tool

Meeting Accessibility Design

Your visual bias is your biggest blocker:

  • semantics: word, descriptions, confirmations, errors
  • interactions: animations micro-interactions
  • place: location on screen, movement, meaning from states (like hover on/off)

Other bias:

  • mouse heavy bias (requires mouse to use easily) - a lot of user only use their keyboard
  • colour only bias (error message only in red text)

Design phase, think of:

  • typography, colour contrast, link visibility and target size, label proximity
  • and then look at: predictable and consistent navigation, visual focus indicator, blocks of content being separated and distinct from each other

Design content by blocks

  • organise content into blocks (grouping)
  • identify the value/purpose of each block
  • determine the best order for blocks to be read be screen reader
  • large blocks will contain smaller block (define each + relationships)
  • document block ordering (for visual design / coding later)

Think accessible layout - determine the following focus points and movements:

  • the order in which elements receive focus
  • the way in which elements are grouped
  • where focus moves when the element in focus disappears
  • clarify where the focus exists through a combination of visual indicators and accessibility text

Prioritising content

  • weigh by Persona need/priority
  • identify your metadata (Headers, Headings, Images)
  • push interruptible elements out of flow: widget (print, share, email etc) or eg cross-promotional stories or tools should not interrupt a story even if they appears in the body of the page
  • structure the flow by context (not visual layout - but by how much sense it makes)
  • develop a quick Accessibility "style guide" early (Blocks, Order, Flow rationale)

Design for focusability

  • the ability to access essential UI elements without a mouse
  • a layout schema for UX designers to map the content priorities on a page
  • a navigation aid to allow smooth flow and reduce audio clutter or unnecessary screen reader overhead (or loopy navigation)
  • CSS keyboard killer, never do: focus {outline:0;}
  • good contrast
  • complementary shape and size to the element
  • colour scheme is complementary, but also stands out
  • doesn't need to be the same for all elements
  • animations help users track focus movement
  • should degrade gracefully (be visible on older browser)
  • should be the same across browsers

Focus design tips:

  • list out focusable elements: How many variations of each do you have? If there are primary and secondary buttons that use different colours, will the same focus indicator work for both of them?
  • think about your colour scheme: Are you using a callout / highlight colour? Would that colour work well for focus? of not, is there a different colour that would?
  • mirror your hover states: in a lot of cases, a good focus state might be the same as the hover state. If your hover state is obvious, consider using that as a focus indication as well
  • custom style your focus: it's generally a good idea to replace the native browser focus style for all elements with a custom style so that focus indicators are consistent across browser

Website and Web application issues and tips

  • assign active images first (CTA, verb/action labels, buttons vs links)
  • then static images and decorative last
  • classic no no: More..., clcik here, learn more, this, links to close to one another
  • rule of thumb: if you're using a keyboard, anyhting you interact with should have a visible focus
  • if using a coloured background, choose one which is single, solid colour, rather than texture or patterned
  • the colour of your background does not matter as much as the contrast between the background and the text
  • make sure your chosen colour scheme may be over-ridden by the browser settings if necessary
  • avoid italic text as this can appear 'wobbly' on screen
  • capitalisation of a sentence should be avoided
  • primary things should be visible, secondary should be shown on hover - note that Hovers can be distracting for users with cognitive or mobility impairment
  • small texts should have a contrast ratio of at least 4.5:1 against its background
  • large texts (at 14pt bold / 18pt regular and up) should have a contrast ratio of at least 3:1 (unless it's a decorative logo)
  • label visual UI elements - this is a common mistake. Screen readers user need to know which UI elements are tappable on screen. To enable screen reader to read the name of components out loud, add the contentDescription attribute to components such as buttons, icons, and tabs containing icons that have no visible text

Mobile Accessibility Issues and Tips

  • same as for desktop: labelling UI, grouping content, creating easy-to-follow navigation, using adequate colour contrast, using cues other than colours
  • making touch target large (44-48px)
  • presenting media content (control, toggle, subtitles or alt format)
  • support large font on apps: iOS use built in Zoom & magnifier feature, on android the font size and font display size only work on Android 7 and up
  • bluetooth keyboard issues: some devices have no physical keyboard, and no easy way to pair one
  • infinte scrolling: furstrating for keyboard users - don't implement it

Accessibility tips for VR/AR

  • avoid height bias (mobility impairements) eg wheelchair, can't reach table. Don't require a lot of body movement
  • avoid visual bias (highlight affordances)
  • avoid cognitive bias (gamers get that you have to play to find UI's, most people do not)

Elderly and Senior Users

  • they tend to read more (rely on instruction for context and cues)
  • scrolling behaviour is limited (unfamiliar)
  • require large fonts, and high contrast
  • are error and mistake phobic
  • get lost when troubleshooting is required
  • take longer to learn new applications or devices
  • take longer to complete tasks
  • use different search strategies
  • perfom worse on tasks relying on memory
  • are more distractible
  • have a harder time dealing with errors
  • make more erratic or accidental movements with the pointer
  • make more input errors
  • have more trouble hitting on-screen targets

7 best practices for inclusive product design

Inclusive Design Thinking mindset

Inclusive design is a methodology that is about valuing a diversity of experiences and abilities in the development of a product as well as in its future use. Designers should recognize that when accessibility issues are an add-on after the product has been developed, it is actually too late.

Reframing the definition of disability from a health condition or personal attribute to, “a mismatch between the needs of an individual and a product, service, social structure, culture, or environment is huge because we all experience these mismatches on a day-to-day level whether they’re permanent, temporary or situational.

Inclusive design methods are a great addition to human-centered design methodologies.

Learning how people adapt to the world around them means spending time understanding their experience from their perspective.
When done well, we can recognise more than just the barriers that people encounter. We also recognise the motivations that all people have in common.

Designing for accessibility is not that hard

Digital accessibility refers to the practice of building digital content and applications that can be used by a wide range of people, including individuals who have visual, motor, auditory, speech, or cognitive disabilities.

There is a strong business case for accessibility: studies show that accessible websites have better search results, they reach a bigger audience, they’re SEO friendly, have faster download times, they encourage good coding practices, and they always have better usability.

  1. add enough colour contrast: According to the W3C, the contrast ratio between text and its background should be at least 4.5 to 1 (conformance level AA.) The ratios become more forgiving with larger and heavier fonts since they’re easier to read at lower contrast. If your type is at least 18 pt or 14 pt bold, the minimum contrast ratio drops to 3 to 1
  2. don’t use colour alone to make critical information understandable: Try to use an indicator other than color such as text labels or patterns. When showing errors on the screen, don’t rely on colored text alone, add an icon or include a title to the message. Consider adding a visual cue such as font weight or underline text style to linked text in a paragraph, so the links stand out.
  3. have clear focus indicators : People with visual and motoric disabilities should be able to navigate without a mouse or trackpad. Created focus indicators that are straight forward and obvious for all so that regardless of your level of ability, you can use a website or apps with clarity.
  4. use labels or instructions with form fields and inputs: Always help people understand what they should do and write in a form. It’s best if labels don’t go away, even when the person is filling an input; people should never lose context with what they’re writing. People who use screen readers usually navigate through a form using the Tab key to jump through the form controls. The <label> elements are read for each form control. Any non-label text, as placeholder text, is usually skipped over.
  5. write useful alternative text for your images and other non-text content: Try to describe what’s happening in the image, and how it matters to the story, rather than just saying something like “picture,” context is everything.
    If the image is purely decorative or if it creates redundancy because the surrounding context already explains the content. Then adding an empty attribute will make screen readers skip it. If you don’t write any alt text, some screen readers will read the file name to the individual. If the image is functional like a printer icon, then give the alt attribute the value of the action that will happen "print", also check Axess Lab guidance about alt text
  6. use correct markup on your content: Headings mark where the content starts — they’re tags given to text to define its style and purpose. Headings also establish the hierarchy of the content of the page.
    Titles with big font sizes help a reader understand the order of the content better. Likewise, screen readers also use heading tags to read content. This way, people with low-vision get an overview of the page by reading each heading in a hierarchal flow.
    It’s important to use proper structural elements when you develop a site. HTML elements communicate to the browser what kind of content they contain and how the browser should render or treat that content. The components and structure of a page are what arranges an accessibility tree. This tree is what powers screen readers which are used by people who are blind so they can “listen” to a page.
  7. use strong typographic hierarchy: good typography is the key to good design. Ensure that typographic hierarchy makes sense without the use of too many different of shades of grays. Rely on strong type sizing and bolding to create a meaningful hierarchy so that information makes sense at a glance. Enabled support for dynamic font sizes. This way you can use the default accessibility feature to change the size of the text to what suits you best.
  8. support keyboard navigation: Keyboard accessibility is one of the most critical aspects of web accessibility. People with motor disabilities, blind people that rely on screen readers, people that don’t have precise muscle control, and even power users are dependent on a keyboard to navigate content.
    The focus state that we discussed before provides a visual indicator of the component that is currently selected.
    As you navigate through a page, the order of the interactive elements is essential, and the navigation must be logical and intuitive. The tab order should follow the visual flow of the page: left to right, top to bottom — header, main navigation, content buttons and inputs, and finally the footer.
    Be careful with the size of your navigation — this includes the number of links and the length of the text. Tabbing through long menus may be demanding for people with motor disabilities. And listening to lengthy links can be cumbersome for people that use screen readers—try to be concise. Providing ARIA landmarks or HTML5 structural elements like <main> or <nav> will make navigation easier.
  9. use SVGs: Replace PNGs with SVGs so that icons and UI components render clear and sharp when you use the built-in zooming feature that’s available on most operating systems and browsers.

These guidelines will help you make your web design get closer to meet level AA of the Web Content Accessibility Guidelines. There are 3 levels A is the lowest, AAA is the higest. More info about level of conformance here.

The developing approach to service building at GOV.UK

The assisted digital team found that usually we default to and encourage service builders to segment users in a way that blocks inclusive service building:

  • digital user: can use an online service independently
  • assisted digital users: can’t and receive help offline to do things online
  • non-digital, non-user: can’t and use a separate, offline way to do things
  • accessibility-related user: with disabilities or impairments

There is also a segmentation by channel:

There are useful things to know when delivering services over specific channels, for example how to handle data securely online. But channels are just the medium between government and citizens, often chosen as a symptom of other factors, for example picking up the phone because a website crashes, or going to a local Citizens Advice because you couldn’t understand what was being said over the phone.
The channel is not the user, and we found that segmenting users by the channel they’re using leads to exclusive design. For example, digital service builders work hard to ensure online content is comprehensible and measurable, but much less hard, or not at all, to make telephone content the same.
Designers do the hardest work to make online stuff simple. Yet the greatest needs and highest costs are around offline interactions.

Inclusive segments for inclusive service building:

We reviewed the reasons why users of all characteristics, on all channels might struggle or fail to successfully use a service, and found that all of us, all users, share areas of limited capacity:

  • time available
  • financial situation
  • ease of physically access to an interface (including a person)
  • interface capability
  • awareness of the service and access options
  • ability to comprehend information
  • mental health / emotional capacity
  • trust in service robustness, security, reliability
  • ability to provide required information
  • self-confidence to be able to do what’s required of them
  • willingness to use the service

We think that approaching guidance around these universally applicable areas will better encourage and enable service builders to build services that are inclusive and work properly for all.

Recite me

From the google accessibility group - Veronika Jermolina :

I tested Recite Me toolbar with users with diverse needs in the past (including dyslexia, vision and motor impairments). The overall recommendation to the client was not to use it as it took control away from users, while not adding any benefits. This was despite us recognising that the intention behind the toolbar was to help as many users as possible.
From memory:

  • Participants didn’t think / know to enable the toolbar as they didn’t see the label or didn't think it applied to them. Participants also didn’t know how to exit the tool once enabled and felt lack of control
  • Participants didn’t know what to expect before clicking / activating a button as icons didn’t make sense (both visually and to screen reader users)
  • The resulting change to the page sometimes made the experience worse (e.g. colour contrast combinations, loss of visual effects like button borders…)
  • There were issues around keyboard access, including enabling the toolbar

Note that Recite Me may have made changes since I tested it about 2 years ago.
My experience is that people with access needs require technologies to help them access all information, not just on one website. Saying that, solutions like Recite Me could work in some instances where users with diverse needs require regular access to a website.

Why GOV.UK content should be published in HTML and not PDF

About PDF

Problems with PDF:

  • most are not accessible
  • simple layout help, simple tagging too!
  • lack of alt text and poor heading are very common
  • like plain English, PDF access is taken for granted
  • outsource PDF markup if you do not have an authoring strategy
  • they need, structure (headings header, footer etc..) metadata (Main lang etc...), navigation, links with description, styles (h1 h2 etc...) images (alt)

Why testing with real user is so important

Once an assistive technology user can find a rhythm that works for them, they tend to stick to it. For Dragon users this often means dictating their speech into a single long Word document, where it is easy to see and edit, and then transferring it into the various dialogue boxes in their web browser.

For JAWS users a dossier of case material prepared and saved into PDF was far easier to read than piecemeal material in various formats under a host of different tabs.

Try to make any content – including lists of options – as concise as possible so that it can be quickly understood.

No one reads every word on every page. We’re looking for the gist. Screen reader users are no different. In fact experienced users often like to speed up the reading rate to 300 words per minute or more. This is their equivalent of scanning the page.

Lots of lines and global navigation create tedium and should allow for easy skipping, ideally in one go, for example with the ‘Skip Navigation’ link.

Links need to go somewhere
It sounds obvious, but with widgets and compartmentalised page content, clicking a link will often update just a section of the page. This is great for sighted users who will spot such a difference. But not for a screen reader user.
There are ways to avoid this issue. You could ensure that the new content is announced by the screen reader by using ARIA live regions or by moving focus to the updated content.

Designers and developers should implement the solution that provides the best user experience. This will depend on the particular design, content and functionality of the page.
Offer a linear and obvious layout
One of the screens we tested had been designed full of embedded, progressive logic. Each time a user answered a question, yet another nested level of questions appeared, depending on how they’d answered the previous question. 3 levels in our screen reader users had no idea which question they were answering anymore.

Include integration with other software such as Word and Adobe Acrobat
Even if your system is web-based, if you are relying on other software, such as Word or PDF, for downloading or viewing material, you must include this in your testing.
In our testing, every Dragon user’s Word software crashed when they tried to download documents. With screen readers, pop-up dialogues to save/open docs in Word were ignored, leaving them stranded.

Problems experienced by assistive technology users are often also experienced by all users
When we tested with assistive technology users, we found them struggling with overly complex screens, missing key pieces of information and struggling to download and open documents.
We found exactly the same thing when we tested with people who weren’t using assistive technology.
Testing with people who use assistive technology can be a quick and effective way of identifying issues that affect all users.

Alternative to PDF, from google group - Fritz Von Runte

Markup:
HTML is a type of markup. A markup is basically a text with notes and rules - indentation, bold (or emphasis), title, quotations, etc.
CSS is a list of styles. It tells the browser the font, font-size, spacing, colours, etc.
For those used to Microsoft Word, the concept of style is not too foreign. It’s just a set of rules that define the layout of the document.
A PDF - when it has text inside - uses a form of markup very similar to HTML called XML. And it uses styles too, although in a way that’s less effective for humans to edit and more effective to maintain every single detail exactly the same, regardless of resolution/size etc…
The reason PDF prints what you see in the screen is because what you see in the screen is a digital form of how it’s going to look on paper. It’s build for paper.
PDF, although being a digital file, is NOT suitable for what some of us call Digital (digital transformation, etc..) Because mainly it is like paper but online.
HTML and CSS can do many many things, like animations for example. But the main thing it tries to be is a legible and flexible text document. Flexible as in it should work not matter how you tried to use it. One can read a HTML in a smartwatch or in a big digital billboard (Piccadilly Circus/Shibuya/Times Square style). PDFs not so much.
HTML and CSS share the responsibilities of making a document accessible.
CSS, for example, will define font size and its proportions (line-height, spacing, etc). Also colours and contrast, all very important for accessibility. HTML helps screen readers, navigation, tips for images and descriptions. Content-wise HTML rules. In terms of how it looks, CSS is the king.

Looking the same on paper as it does on a screen:
If one wants a HTML+CSS combo with very strict rules that will make whatever is on the screen to be transposed exactly the same when printed, that is possible. But that is possibly an under-utilisation of its flexibility.
I’d suggest that if it need to look in a particular way when printed, that this design would be translated to CSS with the tag @media print. This way, when printed the document should look exactly like this particular layout. In parallel to this tag and its rules, the design could have a more flexible set of rules for the screen with the tag @media screen - making it accessible and visible in any device.

Switch Access Scanning

I Used a Switch Control for a Day

Switch Access Scanning is a way of interacting with a computer that requires considerable concentration and physical control. However, those individuals who have certain physical disabilities and dexterity difficulties may depend on this input method.
Switch access scanning tends to be used by those who are not able to operate a touch screen in the traditional way, but are still able to operate a mobile device.
People can use gestures, like the turning of the head, or a hardware bluetooth switch that they can press or even the whole touchscreen can act as a single switch.
The focus of the application moves automatically from one element of the user interface to the next element. When the user wants to perform an action on the element, the switch is pressed and the action is performed.

Accessibility Testing

On Medium - Testing with real people

IDF course - How to design for all - chapter 6

Assistive technology tools you can test with at no cost

Designing UX For People With Disabilities — Case Study

Four Things We Learnt From Facilitating Usability Testing Sessions With Blind Users

Type of testing

  • usability testing: Think Aloud, task based, any device, any location, open discovery of issues
  • accessibility testing: Think Aloud, Assistive Tech (user's device and configurations) Chair's side quite often in their home; open discovery of issues
  • code level testing: testing for code efficiency, errors, guidelines; code de-bugging
  • QA/ User Acceptance testing: testing for what has been designed vs what users need

Users often have a comfort zone within their own settings for assistive technologies, that's why it's important to use their own devices when testing. It's critical as not doing this will affects your results.

For accessibility testing, you follow the same best practice as for usability testing (think aloud, gather metrics, success rate, time on task, errors/ confusions, subjective satisfaction) but also read everything out loud for the users: tasks, subjective satisfaction questionnaire

A.T. by disability type:

  • blind: screen readers, keyboard
  • mobility: speech input, alternative input device (head pointer, single switch device, motion tracking), keyboard
  • cognitive: none
  • Deaf + HOH: closed captioning, speech input
  • low vision: magnifiers, keyboard

What to include in accesibility testing scope:

  • test for keyboard accessibility
  • tab focusability and tab order
  • keyboard functionality
  • keyboard functionality with screen reader
  • make sure you can tab through, go back and move with arrow keys

You also need to be ready, have Braille format if needed, or a sign language interpreter, a friend or family for example.

Formative testing: form your decisions. Goal is to understand issues and problem areas to fix. Use to fix code and to conduct accessible QA.

Summative testing: sums up metrics. Goal is to gather metrics and understand user performance against task success (eg compared to a control group eg users without disabilities). Use to certify compliance or to tease of 3rd party software accessibility.

A major difference with usability testing is because of the poor state of accessibility, users can not always verify or troubleshoot like a user without disabilities. This can lead to accidental skipping or a task or UI element you are trying to have the user evaluate. This means more delicate moderation is needed. "Can you check that last thing again?", "Can you try that one more time for me?". also, things move more quickly and intensely during an accessibility Test, that YOU might be overwhelmed and miss an observation! Use your discretion if conducting summative tests (so you do not bias your data). Good moderation technique is important, as with usability testing.
